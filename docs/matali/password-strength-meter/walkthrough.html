R0 · intent

What is being built

A password-strength meter as a single static web page — vanilla HTML/CSS/JS, no build step, no dependencies — that live-scores a typed password as weak / fair / strong.

The page renders a strength bar, a color-coded label, and a checklist of satisfied rules: length ≥ 8, lowercase, uppercase, digit, symbol. It must be accessible — a labelled input plus an ARIA live region for the strength text — and it deploys as a static page (GitHub Pages, served under a /repo-name/ subpath).

The framing that threads the needle

NIST SP 800-63B discourages composition rules as a mandatory gate but endorses a strength meter as user guidance. So the five-rule checklist is presented as “here’s what you’ve satisfied” — a composition guide — not as a security requirement.

R0 · constraints & floors

Five binding floors at R0

These are the non-disarmable constraints carried out of R0. Every downstream phase inherits them; a build that violates any floor fails the gate. P1 SPEC later formalized the set to eight (see the 8-floor block). Click a floor for its rule text and source.

Why these are floors, not preferences

Each traces to an external authority — WCAG success criteria, the W3C WAI password example, NIST 800-63B, or the static-deploy reality of GitHub Pages. They are the hard requirements stable across every research axis, so they bind rather than negotiate.

R0 · key decision

Composition rules over entropy estimation

The load-bearing choice of R0: score the password by additively counting satisfied composition rules, rather than estimating crack-time with an entropy library. The no-dependencies floor decides it — zxcvbn / zxcvbn-ts cannot ship.

CriterionComposition rules · chosenEntropy estimation (zxcvbn)
No runtime dependency✓ ~10 lines native JS— ~400–800 KB bundle
Zero network calls✓ inline— CDN tag breaks floor
Per-keystroke cost✓ sub-millisecond↑ heavier on old devices
Detects P@ssword1 as weak— rewards it 5/5✓ dictionary / pattern aware
Honest about its limits↑ via UI copy✓ models guessability

The trade-off is explicit and accepted: composition scoring is dependency-free and fast but blind to guessability. The mitigation is honesty in the UI copy, not a heavier engine. The R0 briefing collapsed the rule count into three bands; P1 SPEC later split out an explicit Neutral state and added a length-12 Strong gate (see the band mapping).

The honest limitation, stated up front

Composition-only scoring rewards predictable patterns: P@ssword1 satisfies all five rules and scores Strong on a naive meter, yet is trivially crackable. The label vocabulary must avoid “secure” / “safe” and the copy must disclose that the meter checks composition — not entropy, breach exposure, or guessability. P1’s length-12 gate blunts this by demoting Password1! to Fair.

R0 · risks & anti-patterns

What the meter cannot promise

The briefing documents the risks honestly. None is a reason to abandon the approach — each is a reason to label limitations in the UI copy and constrain the spec downstream.

! Composition rules nudge users toward predictable patterns›
Five rules produce predictably formatted passwords, not strong ones — users append 1!, capitalize the first letter, or substitute @ for a, all known attacker heuristics. NIST 800-63B-4 documents this directly. Mitigation: frame the checklist as “satisfied rules” not “requirements,” and lead with a length-first message.
research.md · Risks
! False sense of security from composition-only scoring›
Without a dictionary / breach-corpus check, P@ssword1 scores Strong on any naive composition meter. The meter cannot distinguish Tr0ub4dor&3 from Password1!. It is a UX affordance, not a security control — downstream consumers (a future auth flow) must not use its output as a gate.
research.md · Risks
+ Color-only indication fails WCAG 1.4.1›
A red / yellow / green bar is illegible to users with red-green colorblindness (~8% of males). Under SC 1.4.1, color must not be the sole signal. The text label is the primary indicator; the bar is decorative reinforcement. (This is also the color-plus-text floor.)
research.md · Risks
+ aria-live over-announcing on every keystroke›
aria-live="assertive" on a per-keystroke region interrupts the screen reader destructively. polite (or role="status") defers until idle. Mitigation: keep the region minimal (band label only), set aria-atomic="true", and never expose the checklist items as live regions. An optional 150–300 ms debounce calms announcement frequency — not throughput. (P1 fixed this at a 200 ms trailing debounce.)
research.md · Risks
+ Strength labels can be gamed or misread as a gate›
“Strong” in green is psychologically powerful — users and integrating developers may treat it as clearance to proceed. The label vocabulary must reflect composition state only and must not use “Secure.” A note that the meter checks composition rules, not breach exposure, is the responsible disclosure.
research.md · Risks
· Performance is not a real risk at this scale›
Five regex checks on a short string are O(1) per rule and well under 1 ms in any modern browser. No architectural mitigation is warranted for a vanilla single-page implementation.

P1 · spec intent

The spec’s intent: educational nudging, not a guarantee

P1 SPEC restated the value precisely: live, accessible, client-side guidance as the user types — a strength band, a color-and-text bar, and a checklist of which composition rules are satisfied.

The value is educational nudging toward longer, more varied passwords — explicitly framed as a composition guide, not a security guarantee. NIST SP 800-63B discourages composition rules as a gate but endorses a strength meter as guidance, with length as the primary signal. It runs entirely in the browser, transmits and stores nothing, and deploys as a single static page to GitHub Pages.

The spec inherits R0’s decision as a fixed input

The composition-over-entropy choice and the floors landed at R0 are not re-litigated in P1 — they are consumed as settled. The spec’s job is to turn the chosen direction into a testable contract: an approach, a band mapping, 11 acceptance criteria, and boundaries.

The chosen approach (options level — the file-by-file blueprint is P2’s job): a clean, layered single-page meter with the scoring core in an importable module. Three files, each a separate concern.

Purity is the privacy floor, expressed structurally

A pure scorePassword — no DOM, no globals, no network, no logging — cannot reach fetch/console/storage, so it cannot leak the password. The rejected alternatives are explicit: an entropy engine (zxcvbn) is excluded by the no-dependencies floor; a single inline <script> would block node --test from unit-testing the scorer; a multi-file CSS/JS split adds relative-path surface for no benefit. The pure-core-in-a-module shape is the smallest structure that is both deploy-safe on the /matali-fullpipeline-dogfood/ subpath and unit-testable.

P1 · band mapping

Four bands, and the length-12 gate

The band mapping is total and unambiguous. The length-12 Strong gate weights length per NIST and blunts the Password1! failure mode — making it Fair, not Strong.

BandCondition
Neutralempty input — no band, no announcement, bar reset
Weaklength < 8  OR  (length ≥ 8 AND ≤1 character class)
Fairlength ≥ 8 AND 2–4 character classes, but not Strong
Stronglength ≥ 12 AND all 4 character classes

Two worked examples carried into the UI copy

Password1! (9 chars, all 4 classes) → Fair — not Strong, blocked by the length-12 gate. correcthorsebatterystaple (25 chars, 1 class) → Weak — the openly-disclosed passphrase limitation of composition-only scoring. The four character classes are lowercase, uppercase, digit, and symbol.

P1 · acceptance criteria

Eleven testable criteria

Each criterion carries a verify hint naming its instrument — node for the pure scorer and source-token gates, browser for DOM and live-region behavior. These are the executable contract every later phase is checked against. Click to expand.

1 Empty input is Neutral›
With the field empty (or after clearing), no band is shown, the bar is reset to empty/neutral, and the live region is empty — no stale “Strong” announcement persists.
verify · browser: clear field → bar empty; live region textContent empty/neutral
2 length < 8 is Weak›
Any input under 8 characters shows the text label “Weak” regardless of character variety.
verify · browser: type 'Ab1!' → label text == Weak
3 Mid-variety at length ≥ 8 is Fair›
An 8+ character input with 2–3 character classes (and any 8–11 input even with all 4 classes) shows “Fair”.
verify · browser: type 'abcdefg8' → Fair; type 'Password1!' → Fair
4 Long + full-variety is Strong›
A length ≥ 12 input with all 4 character classes shows “Strong”.
verify · browser: type 'MyDogR3x!2024' → label text == Strong
5 Password1! is Fair, not Strong›
The length-12 gate is observable: a 9-char all-class input does not reach Strong.
verify · node: scorePassword('Password1!').band === 'fair'
6 Live checklist of 5 rules›
The checklist renders all five rules (length ≥ 8, lowercase, uppercase, digit, symbol) framed as “satisfied rules”, each toggling satisfied/unsatisfied live on every keystroke matching scorePassword(...).rules.
verify · browser: type 'aB3$' → lower+upper+digit+symbol satisfied, length unsatisfied
7 Color is never the sole signal; label is explicit›
The band is conveyed by the visible text label (primary) in addition to bar color; the <input type="password"> has an explicit <label for> with a matching id (no placeholder-as-label).
verify · browser: label[for=pw] references the input id; band text node present alongside the bar
8 Accessible live announcement, value never leaked›
A single role="status" (or aria-live="polite") aria-atomic="true" region announces only the band phrase (e.g. “Password strength: Fair”) and never the password value or any rule text containing it.
verify · browser: live region uses role=status/aria-live=polite; text never equals the typed value
9 Zero exfiltration›
No fetch/XHR, no console.* of the value, no localStorage/sessionStorage writes; the password value never enters the live region or any readable attribute.
verify · node: score.js source contains no fetch/console/localStorage tokens
10 Honest framing›
A persistent visible disclaimer states the meter checks composition only — not entropy, breach exposure, or guessability; UI copy never uses the words “secure” or “safe”.
verify · browser: disclaimer present; body text contains neither 'secure' nor 'safe'
11 scorePassword is pure and unit-tested›
node --test over score.test.js passes, covering each band boundary (Neutral/Weak/Fair/Strong), the length-8 and length-12 thresholds, and each of the 5 rule flags.
verify · node: node --test exits 0; every band + both length thresholds covered
P1 · boundaries

Always, ask-first, never

The spec draws hard edges around the implementer’s discretion: a set of invariants that must always hold, a set of changes that require asking first, and a set of things that must never happen.

always-do
The non-negotiable invariants

Vanilla HTML/CSS/JS only; a pure side-effect-free scorer with node --test coverage (SoC/SRP); sibling-relative asset paths; explicit <label for>; a single polite atomic live region carrying only the band label; color and text for the band; reset to Neutral on empty input; the persistent composition-only disclaimer.

ask-first
Requires explicit approval

Changing the band thresholds (the length-8 / length-12 gates or class counts); adding any file, dependency, build step, CDN script, or web font; editing .github/workflows/pages.yml; changing the band label vocabulary.

never-do
Hard prohibitions

Transmit, log, or persist the password value; use aria-live="assertive"/role="alert" for the per-keystroke region; use the words “secure” or “safe”; use root-relative asset paths (/style.css); add an external library, CDN, bundler, or web font; present the meter as a security gate.

root / final gate dependency edge

The single load-bearing decision, specified exactly

The band logic lives in U1: classCount = number of the 4 character-class rules satisfied. Neutral when length===0; Weak when length<8 OR (length≥8 AND classCount≤1); Strong when length≥12 AND classCount===4; Fair otherwise. satisfiedCount counts all 5 true flags. The scorer touches no DOM, no globals, no network, no logging, no storage — purity is the privacy floor expressed structurally.

Prior art

NIST SP 800-63B stance

NIST explicitly discourages mandatory composition rules as a verifier policy, citing predictable user responses (“password” → “Password1!”). Rev 4 strengthens this to “shall not” impose arbitrary complexity. NIST does recommend offering a strength meter as guidance after a blocklist rejection — so a composition checklist is defensible as user guidance, not a gate. Length is called out as the primary strength signal (8-char floor, 15+ preferred).

  • Frame the five-rule checklist as “what you’ve achieved,” not “requirements.”
  • Weight length over character diversity to align with NIST intent (P1’s length-12 Strong gate does this).
Prior art

Accessible password field pattern

The W3C WAI Complete Password Example is canonical: an explicit <label for> tied to a matching id (no placeholder-as-label), and a semantically separate live region that receives the strength text on each keystroke. The live region uses aria-live="polite" / role="status" — not assertive, which interrupts destructively — and contains only the band label, never the password value.

research.md · Prior Artresearch.md · Existing Implementations
  • The Drupal #521904 case study confirms omitting the live region leaves screen-reader users with no feedback.
  • aria-describedby pointing at the rules list announces them on focus.
Rejected path

Entropy estimation (zxcvbn)

Dropbox’s zxcvbn (and the TypeScript port zxcvbn-ts) detect dictionary words, keyboard walks, date patterns, and l33t substitutions, then estimate crack-time as a 0–4 score. It correctly flags that composition rules “allow weak passwords (like P@ssword1) and disallow strong passwords.” But the minified bundle is ~400–800 KB; even a no-build CDN <script src> tag introduces a runtime dependency — which the no-dependencies floor forbids. Rejected for this feature; its accuracy advantage is the documented cost of the composition model.

U5 — Input wiring + debounced announcement

A thin listener on input#pw: on each input event it reads the value, calls scorePassword(value), and synchronously calls render(result) (checklist + bar + label per keystroke). It separately updates the live region #sr-status to PRESENTATION[band].livePhrase via a 200 ms trailing debounce — the debounce applies ONLY to the live-region text, not to render. On empty value it resets to Neutral and clears the live region, canceling any pending announcement so no stale “Strong” is announced. The value never enters the live region or any readable attribute; no fetch/XHR/console/localStorage anywhere.
