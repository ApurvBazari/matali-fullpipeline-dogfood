# Story — Password-Strength Meter

## R0 RESEARCH

Research surveyed the well-trodden landscape of vanilla, no-dependency password-strength meters and found the patterns have strongly converged. As captured in `research.md`, the dominant approach is a count-of-rules scoring model over five fixed checks (length >= 8, lowercase, uppercase, digit, symbol) mapped to a 3-tier weak/fair/strong bar, with the W3C WAI "Complete Password Example" as the canonical accessibility reference. Research pinned down seven non-disarmable floors (F1–F7): zero runtime and build-time dependencies, the password never leaving the page (no fetch/XHR), an `aria-live="polite"` region present in the initial DOM, color never as the sole signal for strength (text labels and checklist glyphs required per WCAG 1.4.1), static-file deployment that works on both `file://` and GitHub Pages, a programmatically associated `<label>`, and reduced-motion suppression of the bar transition. The central flagged risk is false confidence: NDSS 2017 and USENIX 2023 research show rule-based scoring catastrophically misclassifies passwords like `P@ssw0rd` as "strong", so the meter must be framed as educational guidance and never as a security guarantee. Several choices remain open rather than settled — the exact tier cutoffs (pure rule-count vs. gate-then-count vs. length-aware), the symbol regex (`[^a-zA-Z0-9]` vs. an explicit ASCII set), and the live-region debounce strategy.

## P1 SPEC

The spec, captured in `spec.md`, settled the open questions and committed the feature to a single self-contained `index.html` at the repo root — all CSS and JS inline, no build step, no sibling assets — chosen over a testable two-file split because the deploy workflow uploads the repo root verbatim and one inline file removes every relative-path failure mode. Scoring was locked to a **gate-then-count** tier model: `length >= 8` is a hard gate that pins the tier to Weak until it passes, after which 3–4 of 5 rules earns Fair and all 5 earns Strong, with the bar filling 33% / 66% / 100%. The symbol rule was settled as the broad non-alphanumeric regex `/[^A-Za-z0-9]/` (labelled "special character" in the UI, with accented characters and emoji accepted as documented edge cases), and the live-region strategy as `aria-live="polite"` + `aria-atomic="true"`, present in the initial DOM and updated with only the tier label text on tier change — never per-keystroke and never any password-derived text. The spec records an accepted trade-off: inline logic is not importable, so correctness is verified behaviorally at VERIFY (P4) rather than by unit tests. It closes with **9 acceptance criteria** spanning the labelled input, glyph-plus-text checklist, gate-then-count tiers, bar width and label text, the live region, the symbol regex, the no-network-calls guarantee, WCAG AA contrast with reduced-motion suppression, and a standing educational disclaimer.

## P2 PLAN

The plan, captured in `plan.md`, slices the single inline `index.html` into six dependency-ordered build units that each add a distinct, separately-assertable surface to the same file: **U1** lays down the accessible static scaffold (labelled input, five-rule checklist, bar, tier label, the live region primed with a single space, and the disclaimer); from there **U2** (inline CSS for AA-contrast tier colors, bar geometry, and the reduced-motion guard) and **U3** (the gate-then-count scoring with the five-rule predicates and the `/[^A-Za-z0-9]/` special-char rule) build in parallel; **U4** wires them together with a single synchronous `input` listener that paints the checklist, bar width, and visible tier label per keystroke; **U5** adds the tier-change-only `aria-live` update carrying only the tier string; and **U6** seals the build with a no-network, console-clean check across `file://` and a Pages project-site path, leaving `pages.yml` untouched. Because the single-file choice trades away unit-testable logic, every unit carries a **behavioral** red→green contract — static greps over the file plus headless-browser DOM, computed-style, and live-region assertions — rather than node unit tests. The plan's acceptance map ties all **9 acceptance criteria** back to specific units, and it records that the entire feature is one new file, so rollback is simply removing `index.html`.

## P3 BUILD

The build, captured in `build-summary.md`, landed all six planned units into the single self-contained `index.html` at the repo root — inline `<style>` and `<script>`, zero dependencies — in topological order with **0 micro-cycles** and a 6/6 pass. The opus tier handling U2's tier colors recomputed WCAG AA contrast independently and rejected the `#B7770D` amber the research had leaned toward (only 3.71:1, a fail) in favor of `#8a5a00`, landing all three tier colors at AA (5.44 / 5.93 / 4.72 : 1). The scoring truth table, the render wiring, and the live-region invariants were each independently re-verified at build time — U3's gate-then-count `evaluate()` re-run under node across the tier and special-character cases, U4's `render()` driven under a DOM shim to confirm tier/width/class/glyph and exactly one tier class after a switch, and U5 confirming tier-change-only writes that carry only the tier label and never leak the password. The final unit sealed a zero network surface (11 greps clean, inline-only) and left `.github/workflows/pages.yml` untouched. Per the build summary, end-to-end behavioral verification against the running page remains owned by VERIFY (P4).

## P4 VERIFY

Verification, captured in `verify-report.md`, ran parallel per-criterion probes against the actual running page at `http://127.0.0.1:8753/index.html` and confirmed all 9 acceptance criteria met — but the live, behavioral probing earned its keep by catching two real defects that the static greps and build-time unit checks had missed. The first was **critical**: the strength bar was invisible because `.bar`/`.bar-fill` had no `height` (computed 0px), so the width percentages were set but nothing actually rendered on screen. The second was **medium**: the `aria-live` region carried no `.sr-only` class, so the tier text rendered twice — once in the live region and once in the visible label. One micro-cycle (1 of 3) fixed both with CSS/markup only — an 8px bar track with a full-height fill, and `sr-only` on the live region — leaving the scoring JS, render logic, and live-region write guard byte-unchanged. The fixes were re-verified against the running page with a real browser: computed styles, a clean console, and a screenshot showing a visible green bar at 100% "Strong" with a single on-screen tier label. Five low-severity items were flagged for the human gate rather than fixed — among them emoji length-counting via UTF-16 units, the missing programmatic association from the input to the rules checklist, and a residual non-pristine state after clearing the field — none of them blocking.

## P5 REVIEW

Review, captured in `review-findings.json`, ran a compute-only lens pass: six finders (correctness, spec-adherence, plan-adherence, test-gaps, risk-classify, and engineering-principles) read the diff, then an adversarial refute-by-default verifier re-examined every candidate. The verdict was clean — the plan was fully followed (6/6 units), the risk class came back none, and 8 of 9 acceptance criteria are fully met with AC4 confirmed met once typing begins. Of three substantive candidates, only one survived verification: a low-severity DRY cleanup at `index.html:134`, where `passCount` re-enumerates the same five rule keys already held in `rules` (a single `Object.values(rules).filter(Boolean).length` would be one source of truth). The other two were refuted — the blank-on-initial-render note was rejected as a spec-authorized resting state, and the missing-automated-test note was rejected against the accepted no-framework decision since those paths were already behaviorally verified at VERIFY. No bugs, security issues, risk findings, or floor violations surfaced; the recommended escalation is minor.
